Отлично, вы уже начинаете формировать именно бизнес-логику приложения, а не только технический каркас. Попробую предложить архитектуру и сущности, чтобы потом легче было «натянуть» UI и сеть.

Основные роли

  • Администратор / оператор (сервер)
    Создаёт задания, назначает работников, контролирует ход выполнения, принимает результаты.

  • Работник (мобильное приложение)
    Получает список заданий, выполняет их, ведёт фотофиксацию и отмечает прогресс.

Основные сущности (модель предметной области)

  1. WorkOrder (Работа)

    • id

    • title — краткое название

    • description — описание

    • location — координаты объекта (геопозиция территории/элемента)

    • materials — список расходных материалов (если нужны)

    • assignedTo — идентификатор работника

    • status — состояние (NEW, IN_PROGRESS, DONE, CLOSED)

  2. PhotoRecord (Фотофиксация)

    • id

    • workOrderId

    • type — произвольный тип снимка, не жёстко «до/после», а динамически (BEFORE, AFTER, DEFECT, IN_PROGRESS и т. п.)

    • timestamp

    • geo — координаты на момент съёмки

    • fileUri или remoteUrl

  3. MaterialUsage (Расход материалов)

    • id

    • workOrderId

    • materialId

    • quantity

Логика работы мобильного приложения (для работника)

  1. Работник авторизуется и получает список назначенных WorkOrder.

  2. Выбирает конкретную работу → видит:

    • описание,

    • координаты/объект,

    • список материалов.

  3. Может сделать фотофиксацию:

    • «до начала работ» (тип BEFORE),

    • «обнаружен дефект» (тип DEFECT),

    • «по ходу работ» (тип IN_PROGRESS),

    • «после работ» (тип AFTER).
      👉 Тип фото выбирается из списка, а не жёстко фиксируется шагами.

  4. Отмечает использование материалов.

  5. Отмечает статус работы (IN_PROGRESS → DONE).

  6. Отправляет данные на сервер.

Важный момент: «До» и «После» без жёсткой привязки

То, что вы хотите: гибкая схема фотофиксации.
Решается так: у каждой фотографии есть атрибут type, который задаётся пользователем. Тогда работник может по ходу выполнения сам выбирать:

  • если «снял территорию до начала» → ставит BEFORE;

  • если нашёл проблему → ставит DEFECT;

  • если сделал промежуточный отчёт → IN_PROGRESS;

  • если всё готово → AFTER.

Сервер, администратор или аналитика потом могут группировать фото по типам.

Возможная структура API (упрощённо)

  • GET /workorders → список назначенных работ.

  • GET /workorders/{id} → детали работы.

  • POST /workorders/{id}/photos → загрузка фото (с type и geo).

  • POST /workorders/{id}/materials → списание материалов.

  • PATCH /workorders/{id} → обновление статуса (IN_PROGRESS, DONE).

Архитектура приложения

На клиенте (мобильное приложение) разумно выделить слои:

  1. Data layer

    • Retrofit API для сервера.

    • DataStore/Room для кэша (работы, фото, материалы).

  2. Domain layer

    • UseCase (например: LoadWorkOrdersUseCase, UploadPhotoUseCase).

  3. UI layer

    • Экраны: список работ, детали работы, камера, отчёты.

    • ViewModel для каждой сущности.

🔄 Flow работника

1. Авторизация / вход

  • Работник запускает приложение.

  • Вводит логин/пароль (или авторизуется по токену, выданному администратором).

  • После входа приложение сохраняет токен и получает список назначенных работ.

👉 Экран: LoginScreen → WorkOrdersListScreen

2. Список назначенных работ

  • Отображается список WorkOrder (работ):

    • название,

    • адрес / координаты,

    • статус (NEW, IN_PROGRESS, DONE).

  • Фильтрация: «активные», «завершённые».

  • Работник выбирает одну из работ.

👉 Экран: WorkOrdersListScreen

3. Детали работы

Работник видит:

  • Описание работы (текст).

  • Геопозицию объекта (карта + координаты).

  • Материалы, которые нужны/выданы.

  • Историю фотофиксации (список фотографий с метками: ДО, ПОСЛЕ, ДЕФЕКТ и т. п.).

  • Кнопки действий:

    • «Сделать фото» (с выбором типа: ДО, ПОСЛЕ, ДЕФЕКТ, ПРОМЕЖУТОЧНОЕ).

    • «Отметить материалы».

    • «Начать работу» / «Завершить работу».

👉 Экран: WorkOrderDetailsScreen

4. Фотофиксация

  • Работник открывает камеру → делает фото.

  • После фото появляется диалог выбора типа снимка:

    • ДО начала работ,

    • ДЕФЕКТ,

    • ПРОМЕЖУТОЧНОЕ,

    • ПОСЛЕ.

  • Приложение автоматически сохраняет фото с геолокацией и временем.

  • Фото появляется в списке на экране деталей.

  • При синхронизации отправляется на сервер (POST /workorders/{id}/photos).

👉 Экран: CameraScreen + Dialog выбора типа

5. Учёт материалов

  • Работник может отметить использование расходных материалов.

  • Пример: «Краска — 2 литра, кисть — 1 шт.»

  • Эти данные сохраняются и отправляются на сервер (POST /workorders/{id}/materials).

👉 Экран: MaterialUsageScreen (может быть частью деталей)

6. Управление статусом работы

  • Когда работник начинает работу:

    • Нажимает «Начать работу» → статус становится IN_PROGRESS.

  • В процессе он может добавлять фото и отмечать материалы.

  • Когда работа завершена:

    • Делает фото «ПОСЛЕ» → нажимает «Завершить работу».

    • Статус становится DONE.

  • После проверки администратором на сервере работа может быть закрыта (CLOSED).

👉 Кнопки статуса внутри WorkOrderDetailsScreen

7. Синхронизация

  • Если сеть есть → все фото и данные отправляются сразу.

  • Если сети нет → сохраняются локально (Room), отправляются при восстановлении соединения.

  1. LoginScreen
    ↓

  2. WorkOrdersListScreen
    ↓

  3. WorkOrderDetailsScreen

    • Фотофиксация → CameraScreen + выбор типа

    • Учёт материалов → MaterialUsageScreen

    • Статус работы (кнопки Start/Finish)

Таким образом:

  • Все действия «до», «после», «дефекты» и т. д. не жёстко зашиты в этапы — работник сам указывает тип фото при фиксации.

  • Администратор потом видит полную историю и может анализировать по типам.
